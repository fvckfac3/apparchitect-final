# AppArchitect PRD Validator
**Version:** 1.0
**Status:** Canonical
**Layer:** Validation Framework (Domain Validator)
**Parent engine:** `governance/13-validation-engine-spec.md`
**Rule catalog:** `governance/04-validation-rules.md` (Category 2: Document Validation)
**Standard:** `operational-standards/04-validation-standard.md` (Stage 2 — Schema Validation)
**Pipeline position:** Runs after PRD generation (Stage 9 in master pipeline), before handoff to Technical Spec agent
**Consumes:** `output/06-product-documentation/*-PRD.md`
**Produces:** `output/validation-reports/01-prd-validation-{project_id}-{timestamp}.json`

Purpose
The PRD Validator verifies that every Product Requirements Document generated by AppArchitect is structurally complete, semantically consistent, and free of internal contradictions before it advances to Technical Spec generation or Human Review.
This is the first line of defense against ambiguous, untestable, or unimplementable requirements.
The validator does not judge the quality of the product idea. It judges whether the requirements are buildable, testable, and traceable.
No PRD advances to Stage 7 (Technical Documentation) or Stage 9 (Cross-Reference Validation) without passing this validator with a status of PASS or PASS_WITH_WARNINGS.

Scope
Applies to:
- All PRD types: Master PRD, Feature PRD, User Story Catalog, Acceptance Criteria documents, Business Rules documents
- PRD generated by the PRD agent (per `prompt-contracts/06-prd-agent-prompt.md`)
- PRD regenerated from schema updates
- PRD re-validated after human edits

Out of scope:
- Schema validation (handled by 02-schema-validator.md)
- Cross-document consistency (handled by 03-cross-document-validator.md)
- PRD-to-implementation traceability (handled by 04-consistency-validator.md)
- PRD coverage of all schema features (handled by 05-completeness-validator.md)

Validation Principles
The PRD Validator operates on four irreducible principles:
1. **Every requirement must be testable.** If a requirement cannot be verified by a test, observation, or measurement, it is rejected.
2. **Every requirement must be atomic.** Compound requirements ("the system should do X and Y and Z") must be split.
3. **Every requirement must be traceable.** Each requirement must reference a source: schema field, user discovery input, or tagged assumption.
4. **Every requirement must be implementable.** Requirements describing non-technically-feasible behavior are flagged.

Validation Categories

Category A: Structural Validation
Validates the PRD file is well-formed and parseable.
Checks:
- File exists and is readable
- Valid UTF-8 encoding
- Valid Markdown structure (no unclosed code blocks, no broken headers)
- Frontmatter present and valid (project_id, version, generated_by, generated_at)
- Required top-level sections present (Problem Statement, User Personas, Goals, Non-Goals, Functional Requirements, Non-Functional Requirements, Success Metrics, Open Questions)

Failure mode: CRITICAL — file is not processable.

Category B: Required-Section Validation
Validates that every required PRD section exists and contains substantive content.
Each required section must have:
- A heading (## or higher)
- At least one paragraph of content (no empty sections)
- No placeholder text (TBD, TODO, PLACEHOLDER, Coming Soon, Insert Content Here)

Required sections and minimum content thresholds:

| Section | Minimum Words | Required |
|---------|---------------|----------|
| Problem Statement | 50 | Yes |
| Target Users | 30 | Yes |
| Goals | 20 | Yes |
| Non-Goals | 20 | Yes |
| Functional Requirements | 100 | Yes |
| Non-Functional Requirements | 50 | Yes |
| Success Metrics | 30 | Yes |
| Open Questions | 0 | Optional (recommended) |
| User Stories | 200 | Conditional (if user-facing) |
| Acceptance Criteria | 100 | Conditional (if user-facing) |

Failure mode: ERROR if required section missing; WARNING if section under word count.

Category C: Requirement Quality Validation
Validates that every functional requirement meets atomicity, testability, and clarity standards.

Check C1: Atomicity
A requirement is atomic if it cannot be decomposed into two or more independent requirements.
Invalid example: "The system shall allow users to register, verify their email, and complete a profile."
Reason: Contains three distinct requirements (register, verify email, complete profile).
Action: Split into REQ-001 (register), REQ-002 (verify email), REQ-003 (complete profile).
Failure mode: WARNING (recommend split).

Check C2: Testability
A requirement is testable if it can be verified by a test, observation, or measurement.
Invalid example: "The system shall be user-friendly."
Reason: "User-friendly" is subjective and cannot be objectively measured.
Valid rewrite: "The system shall allow a new user to complete account setup in 3 steps or fewer."
Failure mode: ERROR.

Check C3: Measurability
Quantitative requirements must include a specific numeric threshold.
Invalid example: "The system shall be fast."
Valid rewrite: "The system shall respond to user queries in under 200ms at the 95th percentile."
Failure mode: ERROR if no threshold; WARNING if threshold is vague ("fast", "responsive").

Check C4: Negative Clarity
A requirement that says what the system "shall not" do must specify the boundary of the prohibition.
Invalid example: "The system shall not allow unauthorized access."
Valid rewrite: "The system shall reject all API requests without a valid bearer token, returning HTTP 401."
Failure mode: WARNING.

Check C5: Subject-verb-object completeness
Every functional requirement must have a clear subject (system, user, component), verb, and object.
Invalid example: "Login functionality."
Valid rewrite: "The system shall authenticate users via email and password."
Failure mode: ERROR.

Category D: Identifier and Reference Validation
Validates that every requirement, user story, and acceptance criterion has a unique, well-formed identifier.
Format requirements:
- Requirements: REQ-{NNN} (e.g., REQ-001, REQ-042)
- User stories: US-{NNN} (e.g., US-001)
- Acceptance criteria: AC-{NNN} or AC-{parent_story}-{NNN} (e.g., AC-001, AC-001-01)
- All identifiers within a PRD must be unique
- Identifiers must be referenced in the same form throughout the document
- Identifiers must be referenced at least once (no orphan IDs)

Failure mode: ERROR on duplicates; WARNING on orphans.

Category E: Assumption and Confidence Validation
Validates that all assumptions are explicitly tagged and confidence-scored.
Every assumption must include:
- An `assumptions:` block with a stable identifier (ASM-{NNN})
- A confidence score (0.0 to 1.0)
- A source reference (which schema field, user input, or derivation rule produced it)
- An owner (which agent or human introduced it)

Assumptions with confidence < 0.60 are flagged for human review.
Assumptions that have been converted into requirements without explicit human approval are CRITICAL failures.
Failure mode: WARNING if confidence < 0.60; CRITICAL if low-confidence assumption became a requirement.

Category F: Out-of-Scope and Risk Validation
Validates that the Non-Goals and Risks sections are present and substantive.
The Non-Goals section must enumerate at least 3 items. A Non-Goals section with fewer than 3 items indicates the product team has not adequately considered scope boundaries.
The Risks section (or risk subsection) must enumerate at least 2 items. Each risk must include a likelihood and impact rating.
Failure mode: WARNING if Non-Goals < 3; WARNING if Risks < 2.

Validation Rules
The validator executes the following rules in order. Each rule produces 0 or more findings.

Rule ID: PRD-001
Name: Required-Section Presence
Category: Structural
Severity: ERROR
Trigger: Any required PRD section is missing from the document.
Remediation: Add the missing section with substantive content.

Rule ID: PRD-002
Name: Placeholder Text Detection
Category: Document Quality
Severity: ERROR
Trigger: Section contains placeholder text patterns (TBD, TODO, PLACEHOLDER, "Coming Soon", "Insert Content Here", lorem ipsum).
Remediation: Replace placeholder text with actual content or remove the section.

Rule ID: PRD-003
Name: Untestable Requirement
Category: Requirement Quality
Severity: ERROR
Trigger: A functional requirement contains subjective terms without measurable criteria. Subjective terms include: "user-friendly", "intuitive", "fast", "responsive", "easy", "seamless", "frictionless", "modern", "simple", "beautiful", "powerful", "robust" (without specification).
Remediation: Replace subjective term with a measurable threshold.

Rule ID: PRD-004
Name: Non-Atomic Requirement
Category: Requirement Quality
Severity: WARNING
Trigger: A requirement contains 2 or more conjunctions (and, or, also) connecting distinct actions.
Remediation: Split into separate requirements.

Rule ID: PRD-005
Name: Duplicate Identifier
Category: Identifier
Severity: ERROR
Trigger: Two requirements, user stories, or acceptance criteria share the same identifier.
Remediation: Renumber the duplicate.

Rule ID: PRD-006
Name: Orphan Identifier
Category: Identifier
Severity: WARNING
Trigger: A requirement, user story, or acceptance criterion identifier is defined but never referenced elsewhere in the PRD.
Remediation: Either reference the identifier in the appropriate section or remove the orphaned definition.

Rule ID: PRD-007
Name: Confidence Score Missing
Category: Assumption
Severity: WARNING
Trigger: An assumption is declared without a confidence score.
Remediation: Add a confidence score between 0.0 and 1.0.

Rule ID: PRD-008
Name: Low-Confidence Assumption Promoted to Requirement
Category: Assumption
Severity: CRITICAL
Trigger: An assumption with confidence < 0.60 has been promoted to a functional requirement without explicit human approval.
Remediation: Move the requirement back to the assumptions list with its low-confidence label, or escalate to human review for promotion approval.

Rule ID: PRD-009
Name: Insufficient Non-Goals
Category: Scope
Severity: WARNING
Trigger: Non-Goals section contains fewer than 3 items.
Remediation: Expand the Non-Goals section to explicitly enumerate what is out of scope.

Rule ID: PRD-010
Name: Insufficient Risks
Category: Risk
Severity: WARNING
Trigger: Risks section contains fewer than 2 items OR a risk lacks likelihood/impact rating.
Remediation: Add missing risks or rate each risk on likelihood and impact.

Rule ID: PRD-011
Name: Subjective Terminology
Category: Requirement Quality
Severity: WARNING
Trigger: A requirement uses terms that imply a measurement without specifying one.
Examples: "scalable", "secure", "reliable" without specific thresholds.
Remediation: Add a measurable threshold.

Rule ID: PRD-012
Name: Missing Subject
Category: Requirement Quality
Severity: ERROR
Trigger: A functional requirement lacks a clear subject (system, user, component).
Remediation: Restate the requirement with an explicit subject.

Rule ID: PRD-013
Name: Missing Acceptance Criteria
Category: Testability
Severity: ERROR
Trigger: A user story exists but has no acceptance criteria.
Remediation: Add at least 1 acceptance criterion per user story.

Rule ID: PRD-014
Name: Acceptance Criteria Not Binary
Category: Testability
Severity: WARNING
Trigger: An acceptance criterion cannot be evaluated as pass/fail.
Remediation: Restate the criterion so it is unambiguously true or false at evaluation time.

Rule ID: PRD-015
Name: Requirement Schema Mismatch
Category: Cross-Reference
Severity: ERROR
Trigger: A requirement describes behavior that contradicts the Project Schema.
Remediation: Update either the schema or the requirement, then regenerate dependent documents.

Validation Execution Algorithm

Input
- PRD file path
- Project Schema reference (from session state)
- Configuration (severity override, rule exclusion list)

Process
1. Load PRD file. If unreadable, emit CRITICAL and abort.
2. Parse frontmatter. If invalid, emit CRITICAL and abort.
3. Extract sections. Build section index.
4. Execute Category A (Structural) checks. Halt on CRITICAL.
5. Execute Category B (Required-Section) checks.
6. Extract all requirements, user stories, acceptance criteria. Build identifier index.
7. Execute Category C (Requirement Quality) checks per requirement.
8. Execute Category D (Identifier) checks.
9. Execute Category E (Assumption) checks.
10. Execute Category F (Scope/Risk) checks.
11. Execute Rule ID PRD-001 through PRD-015.
12. Calculate readiness score.
13. Generate validation report.

Output
- validation_report.json (per schema in 13-validation-engine-spec.md)
- findings[] with category, severity, rule_id, source_location, description, remediation
- summary with counts (passed, warnings, errors, critical)
- readiness_score (0-100)

Readiness Score Calculation
readiness_score = (passed_checks / total_checks) * 100
With severity weighting:
- Each ERROR deducts 5 points from readiness
- Each CRITICAL deducts 15 points
- Each WARNING deducts 1 point
- Floor at 0

Pass Thresholds
- PASS: 0 critical, 0 errors, readiness >= 90
- PASS_WITH_WARNINGS: 0 critical, 0 errors, readiness >= 70, < 90
- FAIL_WITH_RECOVERY: 0 critical, 1-3 errors, readiness >= 50
- BLOCKED: any critical OR > 3 errors OR readiness < 50

Validation Result Schema
The validator emits a JSON document conforming to:
{
  "validator": "prd-validator",
  "validator_version": "1.0",
  "project_id": "string",
  "prd_path": "string",
  "run_id": "string (UUID)",
  "run_timestamp": "ISO 8601 string",
  "status": "PASS | PASS_WITH_WARNINGS | FAIL_WITH_RECOVERY | BLOCKED",
  "readiness_score": "integer 0-100",
  "summary": {
    "total_checks": "integer",
    "passed_checks": "integer",
    "warnings": "integer",
    "errors": "integer",
    "critical": "integer"
  },
  "findings": [
    {
      "id": "string (UUID)",
      "rule_id": "PRD-NNN",
      "category": "string",
      "severity": "INFO | WARNING | ERROR | CRITICAL",
      "source_location": {
        "file": "string",
        "section": "string",
        "line": "integer",
        "requirement_id": "string (optional)"
      },
      "description": "string",
      "remediation": "string",
      "auto_fixable": "boolean"
    }
  ],
  "next_actions": [
    "string (human-readable remediation step)"
  ],
  "depends_on_passed": [
    "string (validator IDs that must also pass before advancement)"
  ]
}

Integration Points
This validator is invoked by:
- Master Orchestrator (per `prompt-contracts/01-master-orchestrator-prompt.md` Section 6) after PRD generation stage
- Validation Engine (per `prompt-contracts/02-validation-engine-prompt.md`) at Gate V2 (Document Validation)
- Human Review Interface (per `prompt-contracts/11-human-review-interface-prompt.md`) when a PRD requires human-in-the-loop review

This validator depends on:
- Project Schema (loaded from session state)
- Operational Standard 04 (validation procedure)
- Operational Standard 05 (conflict resolution when finding contradicts existing valid PRD)
- Operational Standard 10 (failure recovery when validator itself fails)

Examples

Example 1: Untestable Requirement Caught
Input PRD excerpt:
"REQ-001: The system shall provide an intuitive user experience."
Finding:
{
  "rule_id": "PRD-003",
  "severity": "ERROR",
  "source_location": { "section": "Functional Requirements", "requirement_id": "REQ-001" },
  "description": "Requirement uses subjective term 'intuitive' without measurable criteria.",
  "remediation": "Replace 'intuitive' with specific UX criteria (e.g., 'completable in 3 steps or fewer').",
  "auto_fixable": false
}

Example 2: Low-Confidence Assumption Promoted
Input PRD excerpt:
"ASM-007: Users prefer AI-generated insights (confidence: 0.45)"
"REQ-042: The system shall display AI-generated insights on the dashboard."
Finding:
{
  "rule_id": "PRD-008",
  "severity": "CRITICAL",
  "source_location": { "requirement_id": "REQ-042" },
  "description": "Requirement REQ-042 derives from assumption ASM-007 with confidence 0.45 (below 0.60 threshold) but has been promoted to a functional requirement without human approval.",
  "remediation": "Either: (a) escalate to human review for explicit promotion approval, or (b) move REQ-042 back to the assumptions list.",
  "auto_fixable": false
}

Example 3: Duplicate Identifier
Input PRD excerpt:
"REQ-005: The system shall..."
"REQ-005: The system shall..."
Finding:
{
  "rule_id": "PRD-005",
  "severity": "ERROR",
  "source_location": { "section": "Functional Requirements" },
  "description": "Duplicate identifier REQ-005 detected.",
  "remediation": "Renumber the duplicate to REQ-006 or higher.",
  "auto_fixable": true
}
