# AppArchitect Schema Validator
**Version:** 1.0
**Status:** Canonical
**Layer:** Validation Framework (Domain Validator)
**Parent engine:** `governance/13-validation-engine-spec.md`
**Rule catalog:** `governance/04-validation-rules.md` (Category 1: Schema Validation)
**Standard:** `operational-standards/04-validation-standard.md` (Stage 2 — Schema Validation, Stage 3 — Required-Fields)
**Pipeline position:** Runs after Project Schema construction (Stage 3 in master pipeline), before Architecture Design (Stage 5)
**Consumes:** `output/02-project-schema/project-schema.json`
**Produces:** `output/validation-reports/02-schema-validation-{project_id}-{timestamp}.json`
**Consumes schema spec:** `schemas/01-project-schema.json`, `schemas/02-architecture-object.json` (for forward-reference checks)

Purpose
The Schema Validator verifies that the Master Project Schema generated by the Schema Agent is structurally complete, internally consistent, and conforms to the canonical JSON Schemas defined in the `schemas/` layer.
This validator is the gatekeeper of the data language. If the schema is invalid, every downstream artifact (architecture, workflows, UX, PRDs, technical specs, exports) is built on a broken foundation.
The validator does not judge whether the product is well-designed. It judges whether the schema correctly describes the product the founder wants to build.
No Project Schema advances to Stage 5 (Architecture Design) without passing this validator with status PASS or PASS_WITH_WARNINGS.

Scope
Applies to:
- `output/02-project-schema/project-schema.json` (the master schema)
- All conditional schema extensions (AI features, monetization, integrations) when applicable
- All schema regenerations (from new discovery inputs, from conflict resolution, from version bumps)

Out of scope:
- Schema-to-architecture consistency (handled by 03-cross-document-validator.md)
- Schema-to-PRD coverage (handled by 05-completeness-validator.md)
- Internal agent contracts (validated separately by prompt-contracts layer)

Validation Principles
The Schema Validator operates on five principles:
1. **Schema conforms to its spec.** Every field, type, and enumeration in the generated schema must match the canonical JSON Schema.
2. **All required fields are populated.** Empty or null values for required fields are failures.
3. **All references resolve.** Every cross-reference (e.g., `feature_id`, `screen_id`, `persona_id`) must point to an existing object.
4. **All enumerations are valid.** Enum values must be from the allowed list.
5. **No circular dependencies.** Entity relationships must be acyclic.

Validation Categories

Category A: JSON Schema Conformance
Validates the generated schema file against the canonical JSON Schemas in the `schemas/` layer.
Schemas used for validation:
- `schemas/01-project-schema.json` (the master)
- `schemas/02-architecture-object.json` (for forward-reference validation)
- `schemas/03-feature.json` (each feature in `features[]`)
- `schemas/04-workflow.json` (each workflow in `workflows[]`)
- `schemas/05-screen.json` (each screen in `screens[]`)
- `schemas/06-persona.json` (each persona in `personas[]`)
- `schemas/07-integration.json` (each integration in `integrations[]`)
- `schemas/08-ai-feature.json` (each AI feature in `ai_features[]`)

Validation:
- All required fields present
- Field types match spec
- Enum values within allowed set
- Array lengths within bounds
- String lengths within bounds
- Pattern matches (for IDs, version strings, etc.)

Failure mode: ERROR per missing field, ERROR per type mismatch, ERROR per invalid enum.

Category B: Required Field Population
Beyond JSON Schema conformance, validates that critical required fields have substantive values (not just empty strings or null).
Critical required fields (per `governance/01-master-project-schema.md` "Schema Completion Requirements"):
- `metadata.project_name` (non-empty, max 80 chars)
- `metadata.project_description` (min 50 chars)
- `metadata.project_type` (from enum)
- `vision.problem_statement` (min 50 chars)
- `vision.proposed_solution` (min 50 chars)
- `personas` (min 1, each with name and description)
- `features` (min 1, each with name and description)
- `workflows` (min 1, each with name and trigger)
- `architecture` (non-null)
- `metadata.success_metrics` (min 1)

Failure mode: ERROR per missing/inadequate value.

Category C: Cross-Reference Resolution
Validates that every cross-reference in the schema points to an existing object.
Reference types to check:
- `feature.workflow_refs` → must exist in `workflows[]`
- `workflow.steps[].screen_ref` → must exist in `screens[]`
- `workflow.steps[].actor_ref` → must exist in `personas[]` or `roles[]`
- `screen.accessible_roles` → must exist in `roles[]`
- `role.permissions[].resource` → must reference a defined resource
- `permission.role` → must exist in `roles[]`
- `ai_feature.dependencies[]` → must reference a schema field, integration, or feature
- `integration.dependencies[]` → must reference a feature, role, or workflow
- `data_model.entities[].relationships[].target_entity` → must exist in `entities[]`
- `architecture.components[].dependencies[]` → must reference another component

Failure mode: ERROR per unresolved reference. Each error must include the broken reference path and the missing target ID.

Category D: Enumeration and Constraint Validation
Validates that all enum values are within allowed sets and all numeric/string constraints are satisfied.
Enumeration sources:
- `metadata.project_stage` ∈ {Ideation, Pre-PRD, Architecture, Build, Beta, Launch, Growth, Scale}
- `metadata.project_type` ∈ {Web, Mobile, API, Desktop, Embedded, Multi-platform}
- `authentication.providers[]` ∈ {Email, Google, Apple, GitHub, Facebook, MagicLink, SSO, Phone, Anonymous}
- `monetization.revenue_model` ∈ {Free, Freemium, Subscription, One-time, Usage-based, B2B licensing, Ad-supported, Marketplace}
- `feature.priority` ∈ {Critical, High, Medium, Low}
- `role.permission_level` ∈ {Read, Write, Admin, Owner}
- `security.compliance[]` ∈ {GDPR, CCPA, HIPAA, SOC2, PCI-DSS, FERPA, COPPA, None}
- `ai_features.prompting_strategy` ∈ {Zero-shot, Few-shot, Chain-of-thought, RAG, Fine-tuned, Multi-agent, Tool-use}
- `architecture.hosting` ∈ {Vercel, Netlify, AWS, GCP, Azure, Railway, Render, Self-hosted, Cloudflare, Fly.io}
- `architecture.database` ∈ {PostgreSQL, MySQL, MongoDB, SQLite, Supabase, Firebase, PlanetScale, Neon, DynamoDB, None}

Failure mode: ERROR per invalid enum value.

Category E: Data Model Integrity
Validates the `data_model` section for relational integrity.
Checks:
- All entity names are unique within `data_model.entities[]`
- All entity names follow naming convention (PascalCase, alphanumeric, max 50 chars)
- All entity fields have names, types, and (where required) constraints
- All relationships reference existing entities
- All foreign key fields have a corresponding target field in the target entity
- No circular foreign key dependencies (e.g., A → B → A)
- Cascade rules specified for all relationships (or explicit "Restrict")
- Indexes defined for all foreign key fields
- Primary key defined for every entity (default: `id` of type `uuid`)

Failure mode: ERROR per circular reference; ERROR per missing primary key; WARNING per missing index.

Category F: AI Feature Governance
Validates the `ai_features` section for safety and governance compliance.
Checks:
- If `ai_features.enabled = true`, all required sub-fields populated
- Each AI feature has explicit inputs, outputs, model, and prompting_strategy
- All AI features have `failure_handling` strategy defined
- All AI features have `confidence_threshold` defined (0.0-1.0)
- All AI features have `escalation_path` defined
- No AI feature lacks a model selection rationale

Failure mode: ERROR per missing safety field.

Category G: Authentication and Authorization Integrity
Validates the `authentication` and `roles`/`permissions` sections for security baseline.
Checks:
- If any `feature[]` has `requires_auth = true`, `authentication.required = true` must be set
- If `authentication.required = true`, at least one provider is specified
- If `authentication.mfa = true`, at least one provider supports MFA
- Every role has a unique ID
- Every role has a non-empty `permissions[]` (or explicit "no permissions" with reason)
- Every permission references an existing role
- No two roles have overlapping `permission_level` for the same resource without explicit justification
- SuperAdmin role, if present, has unique owner and is documented as audit-only

Failure mode: ERROR per missing auth strategy; ERROR per role conflict.

Category H: Integration and External Service Validation
Validates the `integrations` section for completeness and safety.
Checks:
- Each integration has `provider` (string), `purpose` (string), `auth_method` (from enum), `data_exchanged[]` (array of strings)
- Each integration declares `data_classification` (Public, Internal, Confidential, Restricted)
- Each integration with `data_classification` ∈ {Confidential, Restricted} declares `encryption_in_transit = true`
- Each integration declares failure handling (Retry, Circuit-breaker, Fallback, Fail-closed, Fail-open)
- Each integration declares a `rate_limit_strategy`
- Each integration has at least one of: `webhook_endpoint`, `polling_endpoint`, or `sdk_integration`
- All API keys/secrets required are listed in `required_secrets[]` (NOT hardcoded)

Failure mode: ERROR per missing auth method; ERROR per missing failure handling; CRITICAL per hardcoded secret.

Validation Rules

Rule ID: SCHEMA-001
Name: JSON Schema Conformance Failure
Category: Structural
Severity: ERROR
Trigger: Generated schema does not conform to `schemas/01-project-schema.json`.
Remediation: Diff generated schema against canonical; update generated schema to match.

Rule ID: SCHEMA-002
Name: Required Field Missing
Category: Required Field
Severity: ERROR
Trigger: A field marked required in the JSON Schema is missing or null.
Remediation: Populate the field with a substantive value.

Rule ID: SCHEMA-003
Name: Invalid Enum Value
Category: Constraint
Severity: ERROR
Trigger: A field's value is not in the allowed enumeration.
Remediation: Update to an allowed value, or update the enumeration (requires governance approval).

Rule ID: SCHEMA-004
Name: Unresolved Reference
Category: Cross-Reference
Severity: ERROR
Trigger: A reference points to an ID that does not exist in the referenced collection.
Remediation: Either create the referenced object or correct the reference.

Rule ID: SCHEMA-005
Name: Circular Foreign Key
Category: Data Model
Severity: ERROR
Trigger: Entities form a circular foreign key chain.
Remediation: Break the cycle by introducing a junction entity or removing one of the FKs.

Rule ID: SCHEMA-006
Name: Missing Primary Key
Category: Data Model
Severity: ERROR
Trigger: An entity does not declare a primary key.
Remediation: Add an `id` field of type `uuid` to the entity.

Rule ID: SCHEMA-007
Name: Missing Index on Foreign Key
Category: Data Model
Severity: WARNING
Trigger: A foreign key field is not declared as indexed.
Remediation: Add an index declaration for the foreign key field.

Rule ID: SCHEMA-008
Name: Hardcoded Secret
Category: Security
Severity: CRITICAL
Trigger: A secret, API key, or credential appears as a literal value in the schema.
Remediation: Replace with an environment variable reference and add the secret to the `required_secrets[]` list.

Rule ID: SCHEMA-009
Name: Unmotivated AI Feature
Category: AI Governance
Severity: ERROR
Trigger: An AI feature lacks a model selection rationale.
Remediation: Add a `model_rationale` field explaining why the selected model fits the use case.

Rule ID: SCHEMA-010
Name: AI Feature Without Escalation Path
Category: AI Governance
Severity: ERROR
Trigger: An AI feature lacks a defined escalation path for low-confidence outputs.
Remediation: Define the escalation path (human review, fallback, or abort).

Rule ID: SCHEMA-011
Name: Inconsistent Auth Strategy
Category: Security
Severity: ERROR
Trigger: A feature requires auth but the project authentication strategy is undefined.
Remediation: Define the authentication strategy in the `authentication` section.

Rule ID: SCHEMA-012
Name: Duplicate Entity Name
Category: Data Model
Severity: ERROR
Trigger: Two entities share the same name.
Remediation: Rename one entity to maintain uniqueness.

Rule ID: SCHEMA-013
Name: Enum Value Not Declared
Category: Constraint
Severity: WARNING
Trigger: A field uses an enum but the value is not in the canonical enum list AND the schema extension is not documented.
Remediation: Either select a declared value or document the new value in the schema extension notes.

Rule ID: SCHEMA-014
Name: Inadequate Project Description
Category: Quality
Severity: WARNING
Trigger: `metadata.project_description` is fewer than 50 characters.
Remediation: Expand the description to adequately describe the project.

Rule ID: SCHEMA-015
Name: Missing Success Metrics
Category: Quality
Severity: WARNING
Trigger: `metadata.success_metrics[]` is empty.
Remediation: Add at least one measurable success metric.

Validation Execution Algorithm
Input
- Schema file path
- Project ID
- Configuration (severity override, rule exclusion list, custom enum extensions)

Process
1. Load schema file. If unreadable, emit CRITICAL and abort.
2. Validate against `schemas/01-project-schema.json` using a JSON Schema validator (Ajv or equivalent).
3. Execute Category A checks programmatically via JSON Schema validation.
4. Execute Category B (Required Field Population) checks.
5. Build index of all IDs in the schema (entity IDs, feature IDs, persona IDs, role IDs, etc.).
6. Execute Category C (Cross-Reference Resolution) checks.
7. Execute Category D (Enumeration) checks against canonical enum tables.
8. Execute Category E (Data Model Integrity) checks. Run cycle detection on FK graph.
9. Execute Category F (AI Feature Governance) checks.
10. Execute Category G (Auth/Authz) checks.
11. Execute Category H (Integration) checks including regex scan for hardcoded secrets.
12. Execute Rule ID SCHEMA-001 through SCHEMA-015.
13. Calculate readiness score.
14. Generate validation report.

Output
- validation_report.json (per schema in 13-validation-engine-spec.md)
- findings[] with category, severity, rule_id, source_location, description, remediation
- summary with counts
- readiness_score (0-100)
- generated_schema_hash (for downstream cache invalidation)

Readiness Score Calculation
readiness_score = (passed_checks / total_checks) * 100
With severity weighting:
- Each ERROR deducts 5 points
- Each CRITICAL deducts 15 points
- Each WARNING deducts 1 point
- Floor at 0

Pass Thresholds
- PASS: 0 critical, 0 errors, readiness >= 95
- PASS_WITH_WARNINGS: 0 critical, 0 errors, readiness >= 80, < 95
- FAIL_WITH_RECOVERY: 0 critical, 1-3 errors, readiness >= 60
- BLOCKED: any critical OR > 3 errors OR readiness < 60

Integration Points
This validator is invoked by:
- Master Orchestrator after Project Schema construction
- Validation Engine at Gate V1 (Schema Validation)
- All downstream validators (cross-document, completeness, consistency) depend on this validator passing

This validator depends on:
- Canonical JSON Schemas in `schemas/01-08`
- Operational Standards 04 (validation procedure) and 10 (failure recovery)
- Schema Agent (per `prompt-contracts/10-schema-agent-prompt.md`) for input generation

Examples

Example 1: Unresolved Feature Reference
Input:
{ "workflows": [{ "id": "WF-001", "steps": [{ "feature_ref": "FEAT-999" }] }] }
Findings:
{
  "rule_id": "SCHEMA-004",
  "severity": "ERROR",
  "source_location": { "json_path": "$.workflows[0].steps[0].feature_ref", "field": "feature_ref" },
  "description": "Reference FEAT-999 does not exist in features[].",
  "remediation": "Add FEAT-999 to features[] or change reference to an existing feature ID."
}

Example 2: Hardcoded Secret
Input:
{ "integrations": [{ "provider": "Stripe", "api_key": "sk_live_abc123xyz" }] }
Findings:
{
  "rule_id": "SCHEMA-008",
  "severity": "CRITICAL",
  "source_location": { "json_path": "$.integrations[0].api_key" },
  "description": "Hardcoded secret detected in integration configuration.",
  "remediation": "Replace literal value with environment variable reference: ${STRIPE_API_KEY}. Add to required_secrets[]."
}

Example 3: Circular Foreign Key
Input:
entities: [
  { "name": "User", "fields": [{ "name": "team_id", "fk_to": "Team.id" }] },
  { "name": "Team", "fields": [{ "name": "owner_id", "fk_to": "User.id" }] }
]
Findings:
{
  "rule_id": "SCHEMA-005",
  "severity": "ERROR",
  "source_location": { "json_path": "$.data_model.entities" },
  "description": "Circular foreign key dependency detected: User → Team → User.",
  "remediation": "Introduce a junction entity (e.g., TeamMembership) or remove one of the FKs."
}

Ownership
The Schema Validator is maintained by:
- Validation Engine (executes)
